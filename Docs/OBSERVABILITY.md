## 1. Overview

The observability stack provides three pillars: metrics, alerting, and logging.

```
App ──/metrics──► Prometheus ──► Alertmanager ──► alertmanager-discord ──► Discord
 │                    │
 │                    ▼
 │               Grafana (:3000)
 │
 └──/logs──► Operator (HTTP)
 └──stdout──► docker logs
```

| Pillar | Tool | Access |
|--------|------|--------|
| Metrics | Prometheus + `prometheus-flask-exporter` | `:9090`, `/metrics` |
| Alerting | Alertmanager + Discord webhook | `:9093` |
| Dashboards | Grafana | `:3000` |
| Logging | `python-json-logger` + LogBuffer | `docker logs`, `/logs` |

## 2. Application Metrics

Metrics are exposed at `GET /metrics` in Prometheus text exposition format. The
endpoint is registered by `prometheus-flask-exporter` in `app/monitoring.py`.

```bash
curl http://localhost/metrics
```

### Metric Families

| Family | Metric Examples | Type |
|--------|-----------------|------|
| HTTP request duration | `flask_http_request_duration_seconds_bucket`, `_sum`, `_count` | histogram |
| HTTP request total | `flask_http_request_total{method, status}` | counter |
| Application info | `app_info{version="0.1.0"}` | info |
| Python GC | `python_gc_objects_collected_total`, `python_gc_collections_total` | counter |
| Process | `process_virtual_memory_bytes`, `process_resident_memory_bytes`, `process_cpu_seconds_total`, `process_open_fds` | gauge/counter |

### Self-Exclusion

The `/metrics` endpoint is excluded from its own tracking
(`excluded_handlers=["/metrics"]`) to prevent feedback loops where scraping
inflates request counts.

### Testing Mode

When `app.config["TESTING"]` is `True`, `PrometheusMetrics` is not initialized.
This prevents Prometheus registry conflicts when multiple test fixtures
create application instances.

## 3. Prometheus Configuration

**File:** `prometheus/prometheus.yml`

```yaml
global:
  scrape_interval: 5s
  evaluation_interval: 5s

rule_files:
  - "alerts.yml"

alerting:
  alertmanagers:
    - static_configs:
        - targets:
          - 'alertmanager:9093'

scrape_configs:
  - job_name: 'flask_app'
    static_configs:
      - targets: ['app:5000']
    metrics_path: '/metrics'

  - job_name: 'prometheus'
    static_configs:
      - targets: ['localhost:9090']
```

### Scrape Targets

| Job | Target | Path | Interval |
|-----|--------|------|----------|
| `flask_app` | `app:5000` | `/metrics` | 5s |
| `prometheus` | `localhost:9090` | `/metrics` | 5s (self-monitoring) |

Prometheus resolves `app` via Docker DNS. When replicas are enabled, Docker DNS
returns all container IPs but Prometheus scrapes only the first resolved
address. For per-replica scraping, additional service discovery configuration
would be required.

## 4. Alert Rules

**File:** `prometheus/alerts.yml`

Four alert rules are defined under the `scalability_alerts` group:

### 4.1 ServiceDown

```yaml
alert: ServiceDown
expr: up{job="flask_app"} == 0
for: 10s
labels:
  severity: critical
```

Fires when the Flask application is unreachable for 10 consecutive seconds.
The `up` metric is automatically generated by Prometheus for each scrape
target.

### 4.2 HighErrorRate

```yaml
alert: HighErrorRate
expr: sum(rate(flask_http_request_total{status=~"5.."}[1m])) / sum(rate(flask_http_request_total[1m])) > 0.1
for: 30s
labels:
  severity: warning
```

Fires when 5xx responses exceed 10% of total requests over a 1-minute window,
sustained for 30 seconds. Uses the `flask_http_request_total` counter exposed
by `prometheus-flask-exporter`.

### 4.3 HighLatency

```yaml
alert: HighLatency
expr: rate(flask_http_request_duration_seconds_sum[1m]) / rate(flask_http_request_duration_seconds_count[1m]) > 1
for: 1m
labels:
  severity: warning
```

Fires when the average request duration exceeds 1 second over a 1-minute
window, sustained for 1 minute. Calculated from the `_sum` and `_count`
components of the HTTP duration histogram.

### 4.4 HighCPU

```yaml
alert: HighCPU
expr: rate(process_cpu_seconds_total[1m]) > 0.9
for: 2m
labels:
  severity: warning
```

Fires when the application process consumes more than 90% CPU for 2 minutes.
Uses the `process_cpu_seconds_total` counter exposed by the Prometheus client
library.

## 5. Alertmanager Configuration

**File:** `alertmanager/alertmanager.yml`

```yaml
global:
  resolve_timeout: 1m

route:
  group_by: ['alertname']
  group_wait: 10s
  group_interval: 10s
  repeat_interval: 1h
  receiver: 'discord_webhook'

receivers:
- name: 'discord_webhook'
  webhook_configs:
  - url: 'http://alertmanager-discord:9094'
    send_resolved: true
```

### Routing Behavior

| Parameter | Value | Meaning |
|-----------|-------|---------|
| `group_by` | `['alertname']` | Alerts with the same name are grouped into a single notification |
| `group_wait` | 10s | Wait 10s after the first alert in a group before sending |
| `group_interval` | 10s | Wait 10s between notifications for a group with new alerts |
| `repeat_interval` | 1h | Re-send notification for unresolved alerts every hour |
| `resolve_timeout` | 1m | Alert is considered resolved if not firing for 1 minute |
| `send_resolved` | true | Send a notification when an alert resolves |

All alerts route to a single receiver: `discord_webhook`, which forwards to
the `alertmanager-discord` bridge service.

## 6. Discord Integration

The `alertmanager-discord` container (`benjojo/alertmanager-discord`) receives
Alertmanager webhook POSTs on port 9094 and forwards them to a Discord channel
via the webhook URL configured in the `DISCORD_WEBHOOK` environment variable.

**Alert flow:**

```
Prometheus evaluates rule
    ↓ (alert fires)
Alertmanager receives alert
    ↓ (groups, deduplicates, waits)
POST http://alertmanager-discord:9094
    ↓
alertmanager-discord formats message
    ↓
POST to Discord webhook URL
    ↓
Message appears in Discord channel
```

Both firing and resolved notifications are sent (`send_resolved: true`).

## 7. Grafana

Grafana (`grafana/grafana-oss:11.5.2`) is deployed on port 3000 with
persistent storage in the `grafana_data` named volume. Configuration:

| Setting | Value |
|---------|-------|
| Admin password | `GRAFANA_ADMIN_PASSWORD` env var |
| User sign-up | Disabled |

Prometheus must be added as a data source manually via the Grafana UI
(`http://prometheus:9090`). Dashboards are configured manually.

## 8. Structured Logging

All application logs are emitted as JSON objects using `python-json-logger`.

**Implementation:** `app/monitoring.py` replaces default Flask and Werkzeug
handlers with a `JsonFormatter`.

**JSON schema:**

| Field | Type | Description |
|-------|------|-------------|
| `asctime` | string | Timestamp (`YYYY-MM-DD HH:MM:SS,mmm`) |
| `name` | string | Logger name (`app`, `werkzeug`) |
| `levelname` | string | `DEBUG`, `INFO`, `WARNING`, `ERROR`, `CRITICAL` |
| `message` | string | Event description |
| `pathname` | string | Absolute path to source file |
| `lineno` | integer | Source line number |

**Access:**

```bash
docker compose logs app
```

Gunicorn access and error logs are directed to stdout/stderr
(`--access-logfile -`, `--error-logfile -`) and captured by Docker's logging
driver.

## 9. Log Buffer and /logs Endpoint

`app/monitoring.py` implements a `LogBuffer` class -- a ring buffer backed by
`collections.deque(maxlen=1000)` that stores the most recent 1000 formatted
log entries in memory.

**Endpoint:** `GET /logs`

| Parameter | Default | Constraint | Description |
|-----------|---------|------------|-------------|
| `limit` | 50 | 1-200 | Number of entries to return |
| `filter` | -- | -- | Case-insensitive substring match across all log fields |
| `level` | -- | -- | Exact match on `levelname` (e.g., `INFO`, `ERROR`) |

**Response:**

```json
{
  "count": 3,
  "logs": [
    {
      "asctime": "2026-04-05 10:00:00,123",
      "name": "app",
      "levelname": "INFO",
      "message": "Monitoring setup complete",
      "pathname": "/app/app/monitoring.py",
      "lineno": 65
    }
  ]
}
```

The buffer does not persist across container restarts. Entries older than the
1000-entry limit are discarded (FIFO eviction).
